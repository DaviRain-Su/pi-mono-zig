# 第15章：设计决策

> 为什么这样设计、权衡、未来

---

## 15.1 为什么只有 4 个基础工具？

### 问题

为什么不内置更多工具？比如：
- `grep` 工具
- `git` 工具
- `npm` 工具
- `docker` 工具

### 决策

只保留 4 个最基础的工具：
- `read` - 读取文件
- `write` - 写入文件
- `edit` - 修改文件
- `bash` - 执行命令

### 理由

**1. 完备性**

通过 `bash`，Agent 可以执行任何命令：
```bash
grep pattern file.txt      # 搜索
git status                 # Git 操作
npm install                # 包管理
docker ps                  # 容器操作
```

**2. 上下文效率**

工具越少，系统提示越短，留给用户意图的上下文越多。

| 方案 | 系统提示大小 | 灵活性 |
|-----|------------|-------|
| 20+ 内置工具 | ~5000 tokens | 固定功能 |
| 4 个基础工具 | ~500 tokens | 无限扩展 |

**3. 自举能力**

Agent 可以用 `write` 创建新工具脚本，用 `bash` 执行。这是 "Agent 构建 Agent" 的基础。

### 权衡

| 优点 | 缺点 |
|-----|------|
| 上下文高效 | 需要更多提示来指导 Agent |
| 完全灵活 | 初学者可能需要学习 |
| 可扩展 | 某些任务需要多步 |

### 实践建议

- 为常用任务创建 Skill，描述如何用 4 个工具完成
- 复杂工作流写成脚本，Agent 直接调用
- 用扩展注册常用工具，按需加载

## 15.2 为什么不是 MCP？

### MCP 的问题

MCP (Model Context Protocol) 设计：
```
下载预构建服务器 → 启动服务器 → 工具常驻上下文
```

问题：
1. **上下文膨胀**：每个 MCP 服务器带 10-20 个工具描述
2. **难以修改**：要改功能需理解服务器代码
3. **静态加载**：会话开始时加载，运行时难变更

### pi 的方案

```
告诉 Agent 需求 → Agent 写代码 → 热重载 → 立即可用
```

### 对比

| 方面 | MCP | pi |
|-----|-----|-----|
| 获取工具 | 下载 | 现场编写 |
| 修改工具 | 改代码 | 告诉 Agent |
| 上下文 | 常驻 | 渐进披露 |
| 协议 | 复杂 | Bash + 代码 |

### 为什么不完全排斥 MCP？

可以通过 `mcporter` 等工具将 MCP 暴露为 CLI，Agent 通过 `bash` 调用。这样：
- 保留 MCP 生态的兼容性
- 不污染 pi 的核心上下文
- 按需使用，用完即走

## 15.3 为什么是会话树？

### 线性历史的局限

传统对话：
```
消息1 → 消息2 → 消息3 → ...
```

问题：
- 尝试新想法会污染历史
- 无法回到之前的状态
- 分支实验难以管理

### 树形结构的优势

```
          消息1
            │
      ┌─────┴─────┐
    方案A       方案B
      │           │
    结果A1      结果B1
                  │
                结果B2（当前）
```

**使用场景**：

1. **方案对比**：尝试 A 和 B 两种实现，比较后选择
2. **实验安全**：大胆尝试，不满意就回退
3. **错误恢复**：走错了路，回到分叉点重新来
4. **并行探索**：多个方向同时探索

### 实现机制

- 每条消息有 `id` 和 `parentId`
- `leafId` 指向当前位置
- `navigate` 不改变历史，只移动指针
- `fork` 创建新文件，复制历史

### 权衡

| 优点 | 缺点 |
|-----|------|
| 安全实验 | 概念比线性历史复杂 |
| 可追溯 | 需要理解树形导航 |
| 可合并 | 存储空间略大 |

## 15.4 为什么是渐进式披露？

### Skill 的设计

Skill 结构：
```markdown
---
name: my-skill
description: 简短描述（常驻上下文）
---

# 完整内容（按需加载）
详细说明、示例、脚本...
```

### 为什么这样设计？

**1. 上下文效率**

- 10 个 Skill，每个描述 100 字 → 常驻 1000 字
- 完整内容可能每个 2000 字 → 按需加载，避免污染

**2. 发现性**

Agent 看到描述，知道"有这个能力"，需要时再加载详情。

**3. 可维护性**

修改 Skill 只需改文件，无需重启，自动生效。

### 与扩展的区别

| | Skill | Extension |
|--|-------|-----------|
| 形式 | Markdown | TypeScript |
| 能力 | 提示工程 + 脚本 | 完整编程 |
| 上下文 | 渐进披露 | 注册时加载 |
| UI | 无 | 可自定义组件 |

## 15.5 为什么是事件驱动？

### 架构选择

pi 采用事件驱动架构：
```typescript
// 核心发布事件
emit('tool_call', { toolName, input });

// 扩展监听并响应
pi.on('tool_call', async (event) => {
  // 拦截、记录、修改...
});
```

### 优势

**1. 松耦合**

扩展不需要知道其他扩展的存在，只关心事件。

**2. 可观察**

所有动作都有事件记录，便于调试和审计。

**3. 可拦截**

扩展可以在事件处理前后插入逻辑（拦截、修改、记录）。

**4. 多消费者**

一个事件可被多个扩展处理，互不干扰。

### 事件类型设计

生命周期事件：`agent_start`, `turn_start`, `message_end`...
工具事件：`tool_call`, `tool_result`...
会话事件：`session_switch`, `session_before_compact`...
请求事件：`before_provider_request`...

### 权衡

| 优点 | 缺点 |
|-----|------|
| 松耦合 | 调试时需要追踪事件流 |
| 可扩展 | 事件顺序可能影响行为 |
| 可观察 | 性能略低于直接调用 |

### 15.5.x 在 Vibe Coding 背景下的思考

在 Agent 自动化盛行、`vibe coding` 成为常态的阶段，为什么“源码分析”仍然是必要动作？

**一句话：让快的行为可控、可审计、可解释。**

#### 1) 事件链路是治理边界，不是“高级用法”

当你用 `/model`、`/compact`、扩展命令、Tool 交互时，核心并不神秘：
- `tool_call` 由谁拦截？
- `tool_result` 是否被改写？
- `before_provider_request` 是否注入了额外上下文？
- `session_before_compact` 是否阻断了危险操作？

理解这些链路后，才能对工程行为设“安全开关”。

#### 2) 速度不是唯一价值

vibe coding 的价值是**快速产生输出**，但工程系统最终受益于：
- 行为可复现
- 失败路径可预期
- 回退与审计路径明确

这三点都来自源代码结构的理解，而不是仅靠 prompt 经验。

#### 3) 让团队可持续协作

团队里最难的是“口径一致”而不是“单点会写 prompt”
- 文档更新和源码实现是否一致？
- 旧签名示例是否在迁移文档中被误用？
- 新扩展是否绕过了既有拦截和会话边界？


#### 4) 可执行实践

每次新增能力前，建议先做 3 个判断：

1. 这个能力是否经过事件层可观测？
2. 是否有完整失败路径（cancel/block/error）？
3. 是否有回退、审计和文档同步机制？

都能回答“是”，再推进自动化规模化；否则先补一轮源码同步。

## 15.6 未来的方向

### 短期（6个月内）

- **更多内置 Provider**：持续跟进新模型
- **改进 TUI**：更好的渲染性能、更多组件
- **扩展生态**：官方扩展仓库、社区分享

### 中期（1年内）

- **多 Agent 协作**：原生支持子 Agent、Agent 间通信
- **更智能的 Skill 发现**：基于上下文自动推荐 Skill
- **可视化会话树**：图形化展示和导航

### 长期（2年+）

- **自适应扩展**：Agent 自动创建、优化、淘汰扩展
- **跨会话学习**：从多个会话中提取模式
- **自然语言编程**：更接近"说人话"的交互

### 不变的原则

无论功能如何扩展，以下原则不变：

1. **最小核心**：4 个基础工具是根基
2. **人在回路**：人类始终掌握最终决策权
3. **可理解**：所有动作可追溯、可解释
4. **开源**：核心永远开源，社区驱动

## 15.7 给贡献者的建议

### 如果你想贡献代码

1. **从小开始**：先修 bug，再写功能
2. **遵循哲学**：新功能是否符合"最小核心"？
3. **测试覆盖**：事件系统复杂，测试很重要
4. **文档同步**：代码和文档一起更新

### 如果你想写扩展

1. **解决自己的问题**：先为自己写一个
2. **分享出来**：发布到社区
3. **接受反馈**：迭代改进
4. **考虑通用性**：能否帮助更多人？

### 如果你想写文档

1. **从用户视角**：他们想知道什么？
2. **提供示例**：代码片段最有价值
3. **保持更新**：与代码同步
4. **翻译贡献**：帮助非英语用户

---

## 本章小结

- **4 个工具**：完备、高效、可自举
- **不是 MCP**：写代码优于下载工具
- **会话树**：安全实验、可追溯
- **渐进披露**：上下文效率与发现性的平衡
- **事件驱动**：松耦合、可观察、可拦截
- **未来**：更多能力，但核心原则不变
