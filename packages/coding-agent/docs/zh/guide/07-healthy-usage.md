# 第7章：健康使用指南

> 避免 Agent Psychosis、质量优先

---

## 7.1 什么是 Agent Psychosis

### 定义

Agent Psychosis（Agent 精神错乱）是指过度依赖 AI Agent，导致：
- 产生低质量代码（"slop"）
- 失去对代码的理解
- 多巴胺驱动的无意义循环
- 对外部贡献缺乏责任感

### 症状

| 症状 | 表现 |
|-----|------|
| 无限循环 | 让 Agent 无监督地反复修改 |
| 代码膨胀 | 产生大量未经审查的代码 |
| 理解缺失 | 不知道代码如何工作 |
| 质量忽视 | 只追求速度，忽视可维护性 |
| 责任转移 | 认为 "Agent 写的，不是我的问题" |

### 根本原因

1. **多巴胺驱动**：快速产生代码带来快感
2. **缺乏现实检验**：没有外部反馈
3. **成本忽视**：不考虑 token 消耗和长期维护
4. **责任模糊**：认为机器可以承担责任

## 7.2 pi 如何帮助你

### 设计上的保护

| pi 特性 | 保护作用 |
|--------|---------|
| **成本显示** | 每次交互显示 token 和费用，让你意识到消耗 |
| **会话树** | 强制你在关键决策点介入（/tree, /fork） |
| **压缩机制** | 限制上下文长度，防止无限累积 |
| **热重载** | 鼓励小步迭代，而非大爆炸式生成 |
| **扩展系统** | 让工具可审计、可修改，而非黑盒 |

### 与 "Slop Loop" 工具的区别

| Slop Loop 工具 | pi |
|---------------|-----|
| 鼓励无限循环 | 强制人在回路 |
| 隐藏成本 | 透明显示 |
| 不可回溯 | 会话树导航 |
| 黑盒生成 | 可审计、可修改 |

## 7.3 健康使用原则

### 原则1：人在回路中

**不要**：
- 让 Agent 无限制地自动运行
- 接受不理解代码
- 忽视审查直接提交

**要**：
- 每个关键决策点人工确认
- 定期使用 `/tree` 回顾历史
- 理解 Agent 生成的代码

### 原则2：质量先于速度

**不要**：
- 追求 "一分钟生成 PR"
- 忽视测试和文档
- 产生 "一次性代码"

**要**：
- 使用 `/review` 分支审查代码
- 让 Agent 解释 "为什么" 而不仅是 "做什么"
- 考虑长期维护

### 原则3：管理成本

**监控指标**：
- Token 使用量
- 累计费用
- 上下文长度

**优化策略**：
- 使用 `/compact` 控制上下文
- 避免无意义的重复提示
- 及时结束已完成任务的会话

### 原则4：对外负责

**向开源项目贡献时**：
- 确保自己理解代码
- 考虑提交 prompt 而非代码
- 尊重维护者的时间
- 接受反馈并改进

## 7.4 实用技巧

### 技巧1：定期回顾

```
# 每30分钟
/tree
# 查看当前位置
# 确认方向正确
```

### 技巧2：小步迭代

```
# 好的方式
"先创建基础结构"
→ 审查
"添加用户认证"
→ 审查
"添加数据库"
→ 审查

# 不好的方式
"帮我做一个完整的应用"
→ 1000行代码
→ 不知道从哪里开始审查
```

### 技巧3：使用分支实验

```
# 重大改动前
/fork
# 在分支上实验
# 满意后回到主线
/tree → 选择分支
```

### 技巧4：记录决策

```
# 在关键节点添加标签
/tree → 按 'l' 添加标签

标签示例：
- "v1-working" - 第一个可用版本
- "before-refactor" - 重构前
- "api-finalized" - API确定
```

### 技巧5：让 Agent 解释

```
# 不要只接受代码
"请解释这段代码的工作原理"
"这个设计的优缺点是什么"
"有没有更简单的方式"
```

## 7.5 识别危险信号

### 🚨 红色警报

- 连续工作超过2小时没有审查
- 产生了超过500行未经审查的代码
- 不知道代码如何工作就提交
- 忽视测试失败

### ⚠️ 黄色警告

- Token 消耗快速增长
- 反复修改同一处代码
- Agent 开始 "猜测" 你的需求
- 上下文接近上限

### ✅ 绿色健康

- 定期使用 `/tree` 回顾
- 每段代码都理解
- 有测试覆盖
- 成本可控

### 每月反思

- [ ] 评估工具使用效率
- [ ] 检查成本是否合理
- [ ] 更新最佳实践
- [ ] 分享经验给团队

## 7.7 团队规范

### 代码审查要求

```
所有 AI 生成的代码必须经过：
1. 作者自己审查
2. 至少一个人工审查
3. 测试通过
4. 文档更新
```

### 提交规范

```
PR 描述必须包含：
- 使用的 prompt
- 关键决策点
- 已知限制
- 测试覆盖情况
```

### 知识分享

```
每周团队分享：
- 有效的 prompt 模式
- 发现的陷阱
- 改进的工作流程
```

## 7.8 在 Vibe Coding 下还要学习源码：为什么？

AI 协助写代码的速度很快，但在 Agent 工程里，“源码理解”不是可选项，而是质量控制能力。

### 一句话理解

- **vibe coding 给你的是速度**
- **源码分析给你的是可控性**
- 两者结合才是稳定生产力。

### 核心价值

1. **减少黑箱风险**
   - 明确知道参数、事件、命令为什么生效。
   - 遇到异常行为，能快速回到源头定位（不是猜测）。

2. **避免文档漂移**
   - CLI 选项、事件名、工具签名是否真实一致，能快速自检。
   - 防止按旧文档/旧示例误写代码。

3. **提高扩展设计质量**
   - 在 `pi.on`、`runner`、`extensions/types` 的层级上，能做出更稳的扩展。
   - 决定“该写 Tool / Command / Event / Skill”的边界更清晰。

4. **团队协作更可持续**
   - 不只生成代码，还能解释“为什么这么做”，便于评审和交接。
   - 便于在团队里沉淀统一规范，而不是靠个人 prompt 技巧。

### 一个可执行原则

在每次启用新能力前，先回答 3 个问题：

1. 我知道它在源码中的真实入口吗？
2. 我知道失败路径（报错、取消、权限、网络）吗？
3. 我知道如何回退、审计和复用吗？

都能回答清楚，就可以大规模自动化；否则先补一轮源码快读。
