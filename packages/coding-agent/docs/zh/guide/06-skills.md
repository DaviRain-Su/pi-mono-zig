# 第6章：Skill 系统

> 使用 Skill、编写 Skill、最佳实践

---

## 6.1 什么是 Skill

### 定义

Skill 是一种**轻量级的能力包**，包含：
- 描述（常驻上下文）
- 详细说明（按需加载）
- 辅助脚本（可选）

### Skill vs Extension

| 特性 | Skill | Extension |
|-----|-------|-----------|
| 形式 | Markdown | TypeScript |
| 能力 | 提示工程 + 脚本 | 完整编程 |
| 上下文 | 渐进披露 | 注册时加载 |
| UI | 无 | 可自定义 |
| 复杂度 | 低 | 高 |

### 使用场景

- **代码审查**：定义审查标准和检查项
- **API 设计**：描述 API 设计原则和示例
- **测试策略**：测试编写指南和模板
- **文档规范**：文档结构和风格指南

## 6.2 使用 Skill

### 查看可用 Skill

启动 pi 时，启动界面显示：

```
Skills: 12 available
```

### 触发 Skill

方式一：Agent 自动识别

当任务匹配 Skill 描述时，Agent 自动加载：

```
你: 帮我审查这段代码
Agent: [自动加载 code-review skill]
```

方式二：手动触发

```
/skill:code-review
```

方式三：带参数

```
/skill:code-review focus=security
```

## 6.3 编写 Skill

### 文件结构

```
~/.pi/skills/my-skill/
├── SKILL.md          # 必需：主文件
├── scripts/          # 可选：辅助脚本
│   └── helper.sh
└── examples/         # 可选：示例文件
    └── example.ts
```

### SKILL.md 格式

```markdown
---
name: my-skill
description: 简短描述（1024字符内），说明这是什么、何时使用
---

# My Skill

## 概述

详细说明这个 Skill 的用途和工作原理。

## 使用步骤

1. 第一步做什么
2. 第二步做什么
3. 第三步做什么

## 示例

### 示例1：基础用法

```bash
./scripts/helper.sh input.txt
```

### 示例2：高级用法

```bash
./scripts/helper.sh --option value input.txt
```

## 参考

- [链接1](https://example.com)
- [链接2](https://example.com)
```

### Frontmatter 字段

| 字段 | 必需 | 说明 |
|-----|------|------|
| `name` | 是 | Skill 名称，小写、字母数字连字符 |
| `description` | 是 | 简短描述，显示在系统提示中 |
| `license` | 否 | 许可证 |
| `compatibility` | 否 | 环境要求 |
| `metadata` | 否 | 任意键值对 |

### 完整示例

**代码审查 Skill** (`~/.pi/skills/code-review/SKILL.md`):

```markdown
---
name: code-review
description: 代码审查指南。使用此 Skill 进行全面的代码审查，包括代码质量、安全性、性能等方面。
---

# 代码审查指南

## 审查清单

### 代码质量
- [ ] 代码是否清晰可读
- [ ] 命名是否规范
- [ ] 是否有适当的注释
- [ ] 是否遵循 DRY 原则

### 安全性
- [ ] 是否有 SQL 注入风险
- [ ] 是否有 XSS 风险
- [ ] 敏感信息是否泄露
- [ ] 输入是否验证

### 性能
- [ ] 是否有不必要的循环
- [ ] 是否有内存泄漏风险
- [ ] 数据库查询是否优化

### 测试
- [ ] 是否有单元测试
- [ ] 测试是否覆盖边界情况
- [ ] 测试是否可维护

## 审查流程

1. **理解上下文**
   - 阅读相关代码和文档
   - 理解业务逻辑

2. **逐行审查**
   - 使用 read 工具查看文件
   - 标记问题点

3. **总结反馈**
   - 分类问题（严重/警告/建议）
   - 提供改进建议

## 输出格式

```
## 审查结果

### 🔴 严重问题
1. [文件:行号] 问题描述
   - 建议：如何修复

### 🟡 警告
1. [文件:行号] 问题描述
   - 建议：如何改进

### 🟢 建议
1. [文件:行号] 问题描述
   - 建议：可选优化
```

## 示例

审查 `src/auth.ts`：

1. 读取文件内容
2. 对照检查清单逐项检查
3. 按输出格式生成审查报告
```

## 6.4 Skill 加载机制

### 渐进式披露

```
系统提示（常驻）
├── Skill 名称和描述（始终可见）
└── 详细内容（使用时才加载）
```

### 加载过程

1. **启动时**：扫描所有 Skill，提取名称和描述
2. **系统提示**：将描述加入上下文
3. **触发时**：Agent 读取完整 SKILL.md
4. **执行**：按照 Skill 指导操作

### 上下文效率

- 10 个 Skill，每个描述 100 字 → 常驻 1000 字
- 完整内容可能每个 2000 字 → 按需加载

## 6.5 Skill 最佳实践

### 1. 描述要具体

❌ 不好的描述：
```yaml
description: Helps with code.
```

✅ 好的描述：
```yaml
description: Performs comprehensive code review including code quality, security vulnerabilities, performance issues, and test coverage. Use when reviewing pull requests or auditing codebase.
```

### 2. 步骤要清晰

Skill 应该像食谱一样，步骤明确：

```markdown
## 使用步骤

1. 读取目标文件
2. 分析代码结构
3. 检查安全问题
4. 生成报告
```

### 3. 提供示例

示例帮助 Agent 理解期望的输出：

```markdown
## 示例输出

```
问题：未处理异常
位置：src/api.ts:45
建议：添加 try-catch 块
```
```

### 4. 保持更新

随着实践改进 Skill：
- 添加新的检查项
- 修正不清晰的描述
- 补充更多示例

### 5. 分享 Skill

将 Skill 放入项目目录，团队共享：

```
project/
├── .pi/
│   └── skills/
│       └── project-specific-skill/
│           └── SKILL.md
├── src/
└── ...
```

## 6.6 辅助资料：Claude Code Skills 经验如何迁移到 pi

> 来源：Thariq 在 Claude Code 团队实践中的总结（2025）
> 原文链接：[https://x.com/trq212/status/2033949937936085378](https://x.com/trq212/status/2033949937936085378)

这篇经验可以直接作为我们理解 pi 的 **Skill 设计** 的补充素材，特别适合回答“该建什么 Skill、怎么建、什么时候共享”。

### 一、先判断 Skill 的用途：借鉴分类模板

Claude Code 的经验里，Skills 常见可分为 9 类。对 pi 而言可按是否“提示驱动”+“脚本驱动”落地：

1. **库与 API 参考**
   - 目标：让模型更快正确使用内部库/API。
   - pi 落地：`SKILL.md` 提供边界、参数示例、Gotchas；必要时放 `references/` 文件。

2. **产品验证**
   - 目标：验证行为是否正确，不止生成。
   - pi 落地：结合 `bash` + 测试脚本（playwright/tmux 等）做可重复验证。

3. **数据获取与分析**
   - 目标：把监控/日志/仓库数据聚合成可复用工作流。
   - pi 落地：把查询规范写在 Skill 文档，复杂逻辑封装到脚本工具。

4. **业务流程与团队自动化**
   - 目标：把重复流程变成单一调用。
   - pi 落地：周报、站会摘要、工单汇总等可放在项目级 `.pi/skills`。

5. **代码脚手架与模板**
   - 目标：统一样板质量。
   - pi 落地：结合 Skill + 脚本模板文件，降低一次性“复制魔改”。

6. **代码质量与审查**
   - 目标：持续约束输出质量。
   - pi 落地：`code-review` 这类 Skill + 自动化检查命令（`pi` 可配 `agent_start`/`tool_call` 事件进行记录）。

7. **CI/CD 与部署**
   - 目标：把部署、回滚、PR 维护流程做结构化。
   - pi 落地：将常用链路拆成多个 Skill 子任务，不要塞进一个超大 Skill。

8. **运行手册（Runbook）**
   - 目标：症状驱动排障（告警→定位→报告）。
   - pi 落地：按服务/错误码/日志口径编写流程化 Skill。

9. **基础设施操作**
   - 目标：高风险操作有防火墙（确认、回滚、边界）。
   - pi 落地：结合“确认步骤 + 限定范围”，并在事件层记录关键参数。

### 二、写 Skill 的 4 条高命中率技巧（对应本文主线）

- **不写显而易见的常识**：重点写“模型容易踩坑的边界”。
- **每个 Skill 都要有 Gotchas**：把最容易错的点、禁用操作、异常分支写出来。
- **把文件系统当作渐进上下文**：`SKILL.md`、`references/*.md`、`scripts/*`、`examples/*` 分层暴露。
- **描述字段按触发器来写**：不是文档摘要，是“何时应选这个 Skill”的索引。

### 三、什么时候共享

适合放入团队共享的 Skill：
- 你会在多个仓库复用；
- 有稳定收益（降低重复沟通或重提错误）；
- 有明确维护者、并能持续补充。

### 四、与 pi 的学习节奏结合

把这篇文章作为**第二层学习输入**，建议和源码同步读：

1. 先看 `guide/06-skills` 了解 pi 的基本机制。
2. 再看本文分类/原则，做你仓库的 Skill 审计。
3. 最后对照代码真值源（`extensions/types.ts`、`extensions/runner.ts`、`slash-commands.ts`）验证行为是否对齐。

> 结论：在 pi 中，Skill 是“让模型记住你团队知识的方法”，而不是“替代工程判断的黑箱工具”。

## 6.7 辅助资料：Prompt Caching 就是一切（Prompt Caching 是系统级优化）

> 来源：Thariq 在 Claude Code 团队实践中的总结（2026/02/20）
> 原文链接：[https://x.com/trq212/status/2024574133011673516](https://x.com/trq212/status/2024574133011673516)

这篇经验的核心结论可以直接应用到 pi：**在多轮对话里，Prompt 缓存不是“优化项”，而是架构约束。**

### 核心理解（用于迁移到 pi）

**1) 缓存是前缀匹配**
- Claude Code 使用 cache-control 前缀复用。
- 任何“前缀变化”（系统提示、工具定义、模型参数等）都会让后续内容失效。
- 结论：把**静态、稳定、可复用**内容放前面，动态内容往后放。

**2) 顺序比内容更多（先静态，后动态）**
- 静态系统提示/工具定义（长期稳定）
- 会话上下文（同会话复用）
- 对话消息（动态、逐轮变化）

**3) 优先用消息更新，避免改系统提示**
- 时间/环境变量这类易变信息，不要硬塞进系统提示。
- 改为在用户消息或 tool_result 中附带更新内容（例如“当前为星期三”“这个文件刚修改过”等）。

**4) 避免对话中切换“模型、工具集”**
- 模型切换会破坏当前对话前缀。
- 在 pi 场景，尽量在单会话内保持模型与可见工具稳定。
- 如果确实需要更轻量/特化模型，可考虑**新会话（如 /fork）**并重建上下文。

**5) 不要中途增删工具**
- 工具集是前缀的一部分。
- 与其移除工具，不如保持工具稳定，透过状态/指令决定是否使用：
  - 示例：使用“计划/只读模式”指令，而非在会话内替换工具。

**6) 压缩（compaction）要“共享前缀”**
- `/compact` 的目标不是“清空历史”，而是“保留可复用前缀”再摘要。
- 实现上可考虑：
  - 保留同一系统提示、工具定义与会话上下文结构；
  - 将压缩请求作为新消息追加，而不是重建一套完全不同的精简 prompt。

### 在 pi 的可执行实践清单

- 在 Skill/Extension 文档里做稳定层次：
  - `README/说明`（稳定）
  - `参数/例子`（可更新）
- 统一输出顺序：避免自动重排序 tool 或脚本列表。
- 对于高频命令，确保触发语句模板稳定，不因时间戳、随机 ID 破坏前缀。
- 把“会话级状态”作为消息承载，减少重载系统层。
- 监控缓存指标：将低命中率当成成本/延迟回归告警指标。

### 迁移要点总结

- **优先级一**：让可复用信息稳定可复用。
- **优先级二**：把波动信息放到消息层。
- **优先级三**：避免会话内结构性变化。

> 结论：构建 pi 的人机工作流时，先做“缓存友好”设计，后做花哨优化。多数成本收益来自“不会被无意打断的前缀”。

## 6.8 辅助资料：使用 Playground 处理非文本问题（可视化交互）

> 来源：Thariq 在 Claude Code 团队实践中的总结
> 原文链接：[https://x.com/trq212/status/2017024445244924382](https://x.com/trq212/status/2017024445244924382)

这篇经验是对“什么时候该用文本、什么时候要上可视化交互”的补充。核心是：当问题更适合画图、拖拽、比较或在空间中编辑时，生成可交互 HTML/页面比长文本更高效。

### 核心要点（可迁移）

1. **用结构化交互替代纯文本说明**
   - 不是替代输出，而是提高语义密度。
   - 对象包括：架构图、组件布局、流程编辑、参数调参、对比评审。

2. **保持可回收的产物**
   - Playground 类输出不是临时截图，而是可独立复用的文件。
   - 在 pi 中可用 `write` 创建 `tmp/*.html`，保留给后续回溯。

3. **把模型反馈变成“可操作问题”**
   - 示例不止是“你觉得怎样”，更像“这里有可点击节点，请给我可接受的改动清单”。
   - 每次交互后再回灌到 skill/agent：把可视化反馈转成下一步指令。

### 如何在 pi 场景迁移

虽然 pi 当前核心是命令行 + 文本交互，但我们可以用同样思路实现：

- 在 Skill 中定义：
  - 什么时候生成可视化（如架构评审、布局对比）
  - 输出文件模板位置（如 `assets/playground/*.html`）
  - 需要回填的决策项（approve/reject/comment）

- 用 `bash` 脚本生成 HTML（或读取模板 + 数据注入），例如：
  - 组件草图
  - 邮件/流程图展示
  - 功能参数面板

- 回到 pi 后，用自然语言让模型对 HTML 产物提出“可执行建议”，并转为文本补丁。

### 示例化迁移场景（改写自原文思路）

- **架构可视化**：让模型输出一个可视化架构页，再对关键节点逐一评论。
- **布局和设计评审**：生成可调参数面板，讨论风格/间距/层级。
- **游戏玩法平衡**：给每个参数设置可视化指标（伤害、冷却、概率）并反复比较。
- **代码结构审查**：在可视化节点上标注“可疑路径/高耦合点”。
- **教学/头脑风暴**：让模型先用可视化组织观点，再产出可执行计划。

### 适合 pi 的最小实践（今天就能上）

1. 先在 Skill 里加入“何时需要可视化”的触发条件。
2. 准备一个 `render-html` 辅助脚本或模版。
3. 产出统一文件名（如 `playground-current.html`）便于追踪对比。
4. 在回合结束前把“评审结论”结构化返回（approve / reject / comment）。

> 结论：文本是默认交互方式，Playground 是当模型表达维度不足时的增益策略。

## 6.10 与 Extension 配合

### Skill 调用 Extension

Skill 可以指导 Agent 使用 Extension 提供的工具：

```markdown
## 使用浏览器工具

1. 使用 browser-tools 的 start 工具启动浏览器
2. 使用 navigate 工具访问目标网站
3. 使用 screenshot 工具截图
```

### Extension 增强 Skill

Extension 可以为 Skill 提供 UI：

```typescript
pi.registerCommand('review', {
  description: 'Start code review with UI',
  execute: async (ctx) => {
    // 显示 UI 面板
    ctx.ui.setWidget('review-panel', [...]);
    // 触发 code-review skill
    ctx.sendUserMessage('/skill:code-review');
  },
});
```

## 本章小结

- **Skill 是轻量级能力包**：Markdown 形式，渐进披露
- **使用方式**：自动识别、手动触发、带参数
- **编写要点**：清晰描述、明确步骤、提供示例
- **最佳实践**：具体描述、保持更新、团队共享
- **与 Extension 配合**：Skill 指导，Extension 增强
