# 第2章：架构故事

> 用餐厅比喻理解五层架构

---

## 2.1 餐厅比喻

想象你走进一家餐厅。这个餐厅有五个层次，每个层次各司其职，但又紧密协作。

### 第一层：厨房设备（基础设施层）

**对应**：Node.js / TypeScript / 文件系统 / 网络

就像餐厅的炉灶、冰箱、刀具——没有它们，什么都做不了。但厨师（和用户）不需要直接操作设备，而是通过更高层的接口使用。

### 第二层：食材供应商（AI 抽象层）

**对应**：`packages/ai`

餐厅需要从不同供应商采购食材：
- Anthropic 提供 Claude（高级和牛）
- OpenAI 提供 GPT（稳定供应的猪肉）
- Google 提供 Gemini（特色海鲜）

供应商不同，但餐厅（pi）用统一的接口采购。这就是 `packages/ai` 做的事情：统一模型接口，无论底层是哪家提供商。

### 第三层：厨师（Agent 核心）

**对应**：`packages/agent`

厨师是餐厅的核心。他们：
- 接收订单（用户输入）
- 决定用什么食材（选择模型）
- 执行烹饪（工具循环）
- 上菜（返回结果）

pi 的 Agent 核心就是这样一个"厨师"：维护状态、执行工具循环、处理事件。

### 第四层：服务员（应用层）

**对应**：`packages/coding-agent`

服务员是顾客和厨房之间的桥梁：
- 记录顾客喜好（会话管理）
- 推荐今日特餐（Skill/扩展系统）
- 处理特殊要求（自定义命令）
- 管理排队等位（消息队列）

这就是 `AgentSession` 的工作：管理会话、加载扩展、处理事件。

### 第五层：餐厅环境（宿主层）

**对应**：`packages/tui` / `packages/web-ui`

餐厅可以是：
- 精致的堂食（TUI 交互模式）
- 外卖窗口（RPC 模式）
- 自助餐厅（Web UI）

同一套厨房（核心），不同的呈现方式。

---

## 2.2 关键洞察：厨师可以用厨房设备创造新菜谱

这是 pi 最核心的设计理念：

> **厨师（Agent）可以用厨房设备（Bash）自己创造新菜谱（扩展）。**

传统软件：
```
顾客 → 服务员 → 厨师 → 固定菜谱 → 上菜
```

pi 的方式：
```
顾客："我想吃融合菜"
厨师："让我创造一个新菜谱..."（用现有设备实验）
厨师："新菜谱已创建，以后可以直接点"
```

这就是**自举能力**：Agent 用 4 个基础工具写代码，创建新工具。

---

## 2.3 数据流：从订单到上菜

### 场景：顾客点了一道定制菜

```
顾客（用户输入）
    │
    ▼
服务员（AgentSession）
    - 记录订单到菜单（会话持久化）
    - 询问今日特餐推荐（加载 Skill）
    - 通知厨房有特殊要求（扩展事件）
    │
    ▼
厨师（Agent）
    - 查看现有食材（AgentState）
    - 决定烹饪步骤（工具循环）
    - 可能需要创造新菜谱（自举扩展）
    │
    ▼
供应商（AI 层）
    - 提供具体食材（LLM 调用）
    - 返回烹饪建议（流式输出）
    │
    ▼
设备（基础设施）
    - 实际执行烹饪（文件读写、命令执行）
    │
    ▼
上菜（结果返回）
    - 顾客看到成品（UI 渲染）
    - 不满意可以重做（分支导航）
```

---

## 2.4 事件系统：餐厅的对讲机

餐厅里，服务员、厨师、供应商通过对讲机沟通。pi 的事件系统就是这样：

```
顾客下单        →  "order_received" 事件
厨师开始烹饪    →  "cooking_start" 事件
厨师需要食材    →  "ingredient_request" 事件
供应商送达      →  "ingredient_arrived" 事件
烹饪完成        →  "dish_ready" 事件
上菜            →  "served" 事件
```

**扩展可以监听这些事件**：
- 经理（扩展）听到 "order_received"，记录到账本
- 质检员（扩展）听到 "dish_ready"，检查是否合格
- 顾客（UI）听到 "served"，看到菜品上桌

---

## 2.5 会话树：菜单的版本控制

传统对话是线性的：
```
消息1 → 消息2 → 消息3 → ...
```

pi 的会话是树形的：
```
          消息1
            │
      ┌─────┴─────┐
    消息2A      消息2B
      │           │
    消息3A      消息3B
                │
              消息4B（当前）
```

**餐厅类比**：
- 顾客点了菜（消息1）
- 厨师做了两个版本尝试（分支 A 和 B）
- 顾客尝了 B 版本，决定继续改进
- 随时可以回到 A 版本，或从消息1重新开始

这就是 `/tree` 命令展示的：你的对话历史是一棵树，可以导航到任意节点，继续或分支。

---

## 2.6 为什么这样设计？

### 分层的好处

| 好处 | 解释 |
|-----|------|
| 可替换 | 换供应商（模型）不影响餐厅运营 |
| 可测试 | 可以单独测试厨师（Agent）不依赖服务员（UI）|
| 可扩展 | 新服务员（宿主层）可以使用同一厨房 |
| 可维护 | 每层职责清晰，问题定位容易 |

### 事件驱动的好处

- **松耦合**：扩展不需要知道其他扩展的存在
- **可观察**：所有动作都有事件记录
- **可拦截**：扩展可以在事件处理前后插入逻辑

### 树形会话的好处

- **实验安全**：尝试新想法不怕破坏历史
- **可追溯**：任何时候都能回到之前的状态
- **可合并**：分支的改进可以带回主线
